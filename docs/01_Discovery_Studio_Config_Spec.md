## Part 1 — ObservationNode Schema

This is the canonical field payload submitted by every surveyor role for every equipment observation. It is flat — no nested objects in the mobile payload. All enrichment and inference metadata is appended post-submission by the backend.

```json
{
  "ObservationNode": {
    "session_id":          "STRING  — UUID, maps to the specific site visit event. Auto-generated.",
    "space_id":            "STRING  — Sticky map label or ad-hoc anchor string. See Part 3.",
    "timestamp":           "TIMESTAMP — Immutable system clock. Auto-generated. Required for forensic sequencing.",
    "observation_type":    "STRING  — UI triage key. Maps to backend equipment_class taxonomy. See Part 4.",
    "fixture_count":       "INTEGER — Physical count of observed load elements. Minimum: 1.",
    "field_state_enum":    "ENUM    — Optional. Observed operational state. Default: UNKNOWN. See Part 5.",
    "did_you_turn_it_off": "BOOLEAN — TRUE if surveyor successfully powered down the asset during the session.",
    "why_not":             "ENUM    — Required when did_you_turn_it_off is FALSE. See Part 6.",
    "surveyor_role":       "ENUM    — Logger identity for audit trailing and role-based analysis. See Part 7.",
    "owl_count":           "INTEGER — Count of after-hours occupants observed in this room. Default: 0.",
    "raw_notes":           "STRING  — Free-text catch-all. Condition markers, anomalies, clarifications.",
    "photo_refs":          "ARRAY OF STRINGS — URIs pointing to uploaded photos. See Part 8."
  }
}
```

### Field-level rules

**`session_id`**
Auto-generated UUID. Never entered by a surveyor. Links every observation to a specific site visit event for session-level grouping and analysis.

**`space_id`**
The canonical location token. Entered by the surveyor using the room code or map label announced by M3. Persists (sticky) across subsequent observations until manually updated. See Part 3 for the full spatial model. Ad-hoc entries are permitted when a space is unmapped — the forensic timestamp engine anchors them post-session.

**`timestamp`**
Immutable. Generated by the system clock at submission time. Never editable. Used for forensic sequencing of ad-hoc `space_id` entries, Owl and Bulldog influence window calculations, and audit trail integrity.

**`observation_type`**
The UI-facing triage dropdown. Collected at the point of capture. The backend maps this immediately to the `equipment_class` taxonomy key used in the Inference Engine. See Part 4 for the mapping relationship. Replaces all prior field names: `asset_sub_class`, `asset_type_enum`, `Measure Type`, `Measure Category`, `Measure Detail`.

**`fixture_count`**
Physical unit count of the observed load under identical conditions. Minimum value: 1. Replaces prior field name `field_count`.

**`field_state_enum`**
Optional. Observed operational state. When logged, it records what the surveyor directly observed — not what the system should be doing. Default value is `UNKNOWN`. Photographs in `photo_refs` and entries in `raw_notes` are the authoritative state signals; `field_state_enum` is a structured shortcut available when the surveyor is confident in the state at point of capture. See Part 5 for permitted values.

**`did_you_turn_it_off`**
Boolean. Records whether the surveyor physically powered down the asset during the session. TRUE only if the surveyor actually operated the control and the asset was confirmed off. If the surveyor attempted but failed, or did not attempt: FALSE.

**`why_not`**
Conditional. Required when `did_you_turn_it_off` is FALSE. Null when `did_you_turn_it_off` is TRUE. The six permitted values map deterministically to Stage 6 control domain allocation in the Inference Engine. See Part 6.

**`surveyor_role`**
Records which role submitted the observation. Used for role-based analysis, coverage validation, and audit trailing. See Part 7 for permitted values.

**`owl_count`**
Integer count of after-hours occupants — people present who are not members of the survey team — observed in this specific room. Applies only to the room where observed (room-scoped). Default: 0. Owl presence modifies confidence scores and H_reduction assumptions in Stage 3 of the Inference Engine. It does not suppress findings.

**`raw_notes`**
Free text. The catch-all for anything that does not fit structured fields. Condition markers ("looks damaged," "leaking"), anomalies, contextual information, and clarifications all go here. Stage 1 of the Inference Engine parses `raw_notes` using bounded keyword matching to flag records for condition inference and `exempt_asset` review. Asset condition (`condition_enum` in prior specs) is not a structured field in the mobile payload — condition information arrives exclusively through `raw_notes`.

**`photo_refs`**
Array of URI strings pointing to uploaded photos. Three named slots are defined (see Part 8). Array may contain 0–3 entries. Photo upload is not required to submit an observation.

## Part 2 — BulldogObservation Schema

A separate, lightweight schema for recording non-occupant workers observed during the session. Submitted via the persistent Bulldog quick-capture control — always accessible, never buried in the observation form.

```json
{
  "BulldogObservation": {
    "session_id":     "STRING    — UUID. Auto-generated. Same session as observations.",
    "timestamp":      "TIMESTAMP — Immutable system clock. Auto-generated.",
    "space_id":       "STRING    — Room or zone where Bulldog was observed. Same input mechanic as ObservationNode.",
    "bulldog_type":   "ENUM      — See Part 9.",
    "count":          "INTEGER   — Number of workers of this type observed. Default: 1.",
    "notes":          "STRING    — Optional. For ambiguous cases, unidentified workers, task context.",
    "badge_photo_ref":"STRING    — Optional. URI pointing to a photo of the worker's badge or name tag."
  }
}
```

**Badge photo:** Photographing the worker's badge or name tag is encouraged when the Bulldog's employer affiliation or role is relevant to the inference (particularly for `Contractor` and `Unknown` types). True Bulldogs — authorized non-occupant workers — should be willing to comply. Refusal to be photographed is itself a signal worth recording in `notes` and flagging for customer awareness. Badge photos are not required for submission.

Bulldog observations feed Stage 3 of the Inference Engine directly. For inference effects, influence window logic, and scope/timeout defaults by type, see `docs/architecture/inference-engine-spec.md` Section 5.3 and Studio Design v3.4 Part 4.

## Part 3 — Spatial Model: space_id and the Sticky Map System

### space_id as the canonical location token

`space_id` is the single location field in the mobile payload. It replaces the prior multi-field location hierarchy (`room_id`, `space_function`, building/floor/zone/room/waypoint fields) in the mobile form. The full location hierarchy is resolved post-session during Stage 6.5 enrichment, not in the field.

**Rationale:** Requiring surveyors to navigate and populate a multi-level location hierarchy during a fast-paced after-hours session introduces errors and slows capture speed. A single sticky token, anchored by M3's verbal announcements and the forensic timestamp engine, produces more reliable location data with less cognitive load.

### The sticky rule

The entered `space_id` persists across all subsequent observations until the surveyor manually clears or updates it. A surveyor entering a new room updates `space_id` once — all observations until the next update inherit that value automatically.

### Ad-hoc entries

When a surveyor encounters an unmapped room or structural anomaly not on the floor plan, they enter a best-guess free-text anchor — for example: `"Next to 101"` or `"Unmapped Mech Pocket"`. The forensic timestamp engine uses the sequential timestamps of surrounding verified observations to anchor ad-hoc entries to the correct location geometry during post-session data cleansing in Stage 6.5.

### M3 coordination

The M3 role is responsible for announcing the current `space_id` verbally when the team enters each zone. Consistent space_id entry across the team depends on M3 accuracy. See `08_Surveyor_Roles` Part 2.

## Part 4 — observation_type to equipment_class Mapping

`observation_type` is the UI triage key — a human-readable dropdown the surveyor selects at point of capture. The backend maps it immediately to the `equipment_class` taxonomy key used in the Inference Engine for routing, wattage lookup, and measure candidate generation.

**Relationship:**
```
observation_type (UI, human-readable)
    → equipment_class (backend taxonomy key, UPPER_SNAKE_CASE)
```

The `observation_type` dropdown is configured per session by the Facilitator. The canonical `equipment_class` taxonomy is maintained in `03_Measure_Catalog_Spec`. This document does not reproduce the taxonomy — it defines the mapping relationship.

**Key rule:** `observation_type` is the only observation category field in the mobile payload. All prior field names (`asset_sub_class`, `asset_type_enum`, `Measure Type`, `Measure Category`, `Measure Detail`) are retired. The backend does not accept them.

## Part 5 — field_state_enum Values

Four permitted values. Case-sensitive. **This field is optional.** Default is `UNKNOWN`.

| Value | Meaning | When to use |
|-------|---------|------------|
| `UNKNOWN` | State could not be determined at point of observation | **Default.** Use when not actively logging a confirmed state. |
| `ON` | Asset is actively energized and operating at point of observation | Surveyor directly confirms the asset is powered and running |
| `OFF` | Asset is confirmed de-energized at point of observation | Surveyor directly confirms the asset is powered down |
| `AUTO` | Asset is under automated control — state may cycle; surveyor cannot confirm ON or OFF | Requires positive identification of a control device (sensor, relay) actively governing this specific asset. Not a fallback for uncertainty — use `UNKNOWN` for that. |

**Notes:**
- `UNKNOWN` is the default and the honest answer when the surveyor did not actively log a confirmed state. It is not a failure — it is the correct value for the majority of observations.
- `AUTO` requires the surveyor to have positively identified a control device governing the specific asset being observed. If the surveyor suspects automation but cannot confirm it, use `UNKNOWN` and note the suspicion in `raw_notes`.
- Photos (`photo_refs`) and `raw_notes` are the authoritative state signals. The Inference Engine reads these first. `field_state_enum` is a structured shortcut that accelerates inference when the surveyor is confident — it does not override photo or notes evidence.
- `field_state_enum` replaces prior field name `state`.

## Part 6 — why_not Enum Values

Four permitted string values plus null. Case-sensitive. Exact strings required — no variations accepted.

| Value | Meaning | Stage 6 routing |
|-------|---------|----------------|
| `"No Switch or Control Found"` | No human-operable control was located during the session — absent or unfound; Enrichment resolves which | `CENTRAL_PANEL_LOOP` (if confirmed absent) or P7 Default Rule (if confirmed unfound) — resolved at Stage 6.5 |
| `"Requires Permission"` | Surveyor could not operate the control due to access or credential limitation | Assessment domain signal — capability check at Stage 5 |
| `"Occupancy Sensor"` | Surveyor positively identified an automated local hardware sweep control actively governing this specific asset | `telemetry_decay_factor` applied in Stage 3 |
| `"Other"` | Ambiguous barrier — surveyor cannot confidently select any of the above; flags for Enrichment Review | Routes to isolated review bucket; stripped from automated savings calculations until resolved |
| `null` | Permitted only when `did_you_turn_it_off` is TRUE | n/a |

**Validation rule:** If `did_you_turn_it_off` is FALSE and `why_not` is null, the payload fails schema validation and is rejected before Stage 1. No exceptions.

**"No Switch or Control Found" — Enrichment resolution:** This single value covers two distinct infrastructure states that cannot be reliably distinguished in the field: (1) no local control exists and the asset is circuit-controlled, and (2) a control exists but was not located during the session. The distinction is resolved during Stage 6.5 Enrichment Review. Until resolved, the Inference Engine applies the P7 Default Rule (reach evaluation forced to NO) as the conservative default.

**"Occupancy Sensor" — positive ID required:** This value requires the surveyor to have directly observed a sensor controlling the specific asset being logged. It is not a default for spaces where automation is suspected. If the surveyor cannot confirm which specific loads a sensor governs, use `"Other"` and describe the ambiguity in `raw_notes`.

**"Other" handling:** Records where `why_not` is `"Other"` are completely stripped from automated savings calculations. They receive a `FACILITATOR_REVIEW_REQUIRED` flag and route to the isolated anomaly bucket in Stage 10. They do not contribute to guarantee or added-value pools until manually resolved by the Analyst during Enrichment Review.

**`why_not` replaces prior field name `why_not_enum`.** Prior values `"No Switch Present"`, `"Control Not Found"`, and `"Timer / Schedule"` are retired. See Part 11.

**V1 scope note:** `"Timer / Schedule"` has been removed from V1. In most field conditions, surveyors cannot reliably distinguish a scheduled asset from one that is simply off without applying judgment. This value is reserved for V2 with additional controls: it will require a `Detail` photo of the control device and will be restricted to `Facilitator` or verified facilities-staff attribution.

## Part 8 — Photo Slots

Three named slots. Slot names are locked. Prior names (`Device Photo`, `Nameplate/Closeup`) are retired.

| Slot | Name | Purpose | Tracks |
|------|------|---------|--------|
| 1 | `Context` | Wide shot establishing spatial context — shows the room or zone, where the asset is within it | Both equipment and condition reading |
| 2 | `Subject` | Full view of the observation subject — the asset itself, clearly centered | Both |
| 3 | `Detail` | Close-up of technically meaningful information — nameplate, control panel settings, thermostat display, breaker label, circuit ID, timer configuration, sensor type | Equipment only |

**Photo capture is encouraged but not required for submission.** An observation without photos is valid. Photo presence increases confidence scores in the Inference Engine and supports Enrichment Review resolution.

**Photo refs in the payload:** Each entry in `photo_refs[]` is a URI string pointing to the uploaded file. The array position does not encode slot — each URI string must be prefixed with the slot name for unambiguous parsing: `"context:https://..."`, `"subject:https://..."`, `"detail:https://..."`.

## Part 9 — bulldog_type Enum Values

Five permitted values. Case-sensitive.

| Value | Scope | Default timeout | Notes |
|-------|-------|----------------|-------|
| `Security` | Whole building | 90 minutes | Confidence reduction building-wide during influence window |
| `Cleaning` | Designated floor | 60 minutes | Confidence reduction on lighting observations on affected floor |
| `Construction` | Target zone | 180 minutes | Loads within scope may be flagged `temporary_load` at Analysis Boundary Gate |
| `Contractor` | Floor | 90 minutes | Same confidence effect as Cleaning — conservative default |
| `Unknown` | Floor | 60 minutes | Conservative defaults until type resolved post-session |

Timeout values are configurable by the Facilitator in pre-session configuration (see Part 10). The defaults above apply when no override is set.

For full Bulldog inference logic — confidence reduction formulas, how influence windows interact with Owl processing, and the `temporary_load` reason code — see `docs/architecture/inference-engine-spec.md` Section 5.3 and Studio Design v3.4 Part 4.

## Part 10 — Facilitator Pre-Session Configuration

The Facilitator configures the following before the session begins. Configuration is locked at the Analysis Boundary Gate. Changes after the gate locks require a new session.

| Configuration item | What it controls | Notes |
|-------------------|-----------------|-------|
| `enabled_fields` | Which ObservationNode fields are active in the mobile UI | All fields are available; Facilitator enables the subset appropriate for this session |
| `required_fields` | Which enabled fields must be populated before submission is accepted | `session_id`, `space_id`, `timestamp`, `observation_type`, `fixture_count` are always required and cannot be disabled. `field_state_enum` is optional and defaults to `UNKNOWN` — it is never required. |
| `observation_type_options` | The dropdown values shown to surveyors for `observation_type` | Configured from the Measure Catalog. Must map to valid `equipment_class` taxonomy keys. |
| `signal_dictionary_version` | Which version of the signal interpretation rules the Inference Engine applies to this session | Locked at session start; must match the catalog version |
| `catalog_version` | Which version of the Measure Catalog is active for this session | Determines `equipment_class` taxonomy and wattage lookup behavior |
| `bulldog_timeout_overrides` | Site-specific timeout values for each `bulldog_type` | Overrides the defaults in Part 9. Used when site-specific knowledge warrants adjustment (e.g., a cleaning crew with a known 2-hour floor cycle) |
| `inference_ruleset_version` | Which version of the Inference Engine rule set governs Stage 6 routing and domain allocation | Defaults to the current production version; overridden only for controlled testing |
